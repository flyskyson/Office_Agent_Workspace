# Idea to Product Skill - 想法落地技能

## 触发关键词

当用户提到以下内容时,自动激活本技能:
- "我有个想法"
- "想添加一个功能"
- "能不能实现..."
- "有个改进建议"
- "新建项目"
- "从零开始做..."
- "不知道怎么做,但想要..."
- 任何**模糊的、未结构化的想法表达**

## 技能描述

本技能实现从**模糊想法**到**可用产品**的完整落地流程,通过结构化的5阶段方法论,确保想法能够快速、系统地转化为实际可用的代码和功能。

## 核心价值

- ✅ **零想法浪费**: 任何想法都能被系统化处理
- ✅ **快速验证**: 1小时内看到可运行的MVP
- ✅ **降低风险**: 通过原型验证避免大量投入后的方向错误
- ✅ **知识积累**: 每个想法都记录在案,形成组织知识库
- ✅ **可复用流程**: 标准化的流程提升效率

## 执行流程

### Phase 1: 理解与澄清 (Clarification) 🎯

**目标**: 将模糊想法转化为清晰的需求定义

**执行步骤**:

1. **识别想法类型**
   ```python
   # 通过对话判断:
   - 新功能 (feature): 添加新的能力
   - Bug修复 (bug): 修复问题
   - 重构 (refactor): 改进代码质量
   - 新项目 (new_project): 从零开始
   ```

2. **引导式对话** - 依次询问以下问题:

   ```
   📋 问题1: 场景描述
   "您能描述一个具体的使用场景吗?"
   → 例如: "张三是个体户,想开业时..."

   📋 问题2: 痛点识别
   "现在的问题是什么?最困扰您的是什么?"
   → 例如: "每次要手动选择模板,耗时且容易出错"

   📋 问题3: 用户定义
   "这个问题影响了谁?"
   → 例如: "市场监管工作人员、办事员"

   📋 问题4: 成功标准
   "理想情况下,您希望如何解决这个问题?"
   → 例如: "上传营业执照后,自动生成申请书"

   📋 问题5: 验收标准
   "如何知道问题已经解决?"
   → 例如: "处理时间从5分钟减少到30秒"
   ```

3. **输出澄清结果**

   ```markdown
   ## 📝 需求定义

   ### 问题陈述
   [清晰描述要解决的问题]

   ### 目标用户
   - [用户类型1]
   - [用户类型2]

   ### 核心痛点
   1. [痛点1]
   2. [痛点2]

   ### 成功标准
   - [可量化的标准1]
   - [可量化的标准2]

   ### 约束条件
   - [约束1]
   - [约束2]
   ```

**Claude操作**:
- 使用 `AskUserQuestion` 工具收集关键信息
- 将对话结果结构化记录
- 调用 `idea_workflow_engine.py` 创建会话

### Phase 2: 探索与分析 (Exploration) 🔍

**目标**: 自动代码探索和可行性评估

**执行步骤**:

1. **自动代码库扫描**
   ```bash
   # Claude自动执行:
   - Grep: 搜索相关代码模块
   - Glob: 查找相关文件
   - Read: 分析现有实现
   ```

2. **分析维度**:
   ```
   🔬 技术可行性
   - 现有代码是否支持?
   - 需要哪些新依赖?
   - 技术难度评估

   🔬 现有解决方案
   - 有无类似功能?
   - 代码复用可能性
   - 最佳实践参考

   🔬 依赖和影响
   - 哪些模块会受影响?
   - 向后兼容性
   - 数据迁移需求

   🔬 风险评估
   - 技术风险
   - 业务风险
   - 时间风险
   ```

3. **输出探索报告**

   ```markdown
   ## 🔍 探索分析报告

   ### 技术可行性
   - 评估结果: ✅ 高 / ⚠️ 中 / ❌ 低
   - 理由: [详细说明]

   ### 现有方案
   - [方案1]: [位置和说明]
   - [方案2]: [位置和说明]

   ### 工作量预估
   - MVP版本: [X小时]
   - 完整版本: [X天]

   ### 依赖项
   - [依赖1]: [说明]
   - [依赖2]: [说明]

   ### 风险与机会
   ⚠️ 风险:
   - [风险1]: [缓解方案]
   - [风险2]: [缓解方案]

   💡 机会:
   - [机会1]
   - [机会2]
   ```

**Claude操作**:
- 使用 `Grep` 搜索关键词
- 使用 `Glob` 查找相关文件
- 使用 `Read` 分析现有代码
- 调用 `IdeaWorkflowEngine.run_phase("exploration")`

### Phase 3: 方案设计 (Design) 📐

**目标**: 生成多个可选方案并对比分析

**执行步骤**:

1. **生成3个方案**:

   ```
   ┌─────────────────────────────────────┐
   │ 方案A - 快速原型 (MVP)              │
   ├─────────────────────────────────────┤
   • 定位: 最小功能,快速验证             │
   • 时间: 1-2小时                       │
   • 适用: 未知方向,需要快速试错         │
   └─────────────────────────────────────┘

   ┌─────────────────────────────────────┐
   │ 方案B - 推荐方案 (平衡)             │
   ├─────────────────────────────────────┤
   • 定位: 平衡功能和开发成本             │
   • 时间: 4-8小时                       │
   • 适用: 需求清晰,追求性价比           │
   └─────────────────────────────────────┘

   ┌─────────────────────────────────────┐
   │ 方案C - 完整方案 (旗舰)             │
   ├─────────────────────────────────────┤
   • 定位: 功能全面,可扩展性强           │
   • 时间: 2-5天                         │
   • 适用: 长期项目,需要持续迭代         │
   └─────────────────────────────────────┘
   ```

2. **详细设计每个方案**:
   ```markdown
   ## 方案X: [方案名称]

   ### 核心思路
   [简要说明]

   ### 技术架构
   ```
   [架构图或说明]
   ```

   ### 实现步骤
   1. [步骤1]
   2. [步骤2]
   3. [步骤3]

   ### 文件变更
   - 新建: [文件列表]
   - 修改: [文件列表]

   ### 优缺点分析
   ✅ 优点:
   - [优点1]
   - [优点2]

   ⚠️ 缺点:
   - [缺点1]
   - [缺点2]

   ### 工作量预估
   - 开发: [X小时]
   - 测试: [X小时]
   - 文档: [X小时]
   ```

3. **方案对比表**:
   ```markdown
   | 维度 | 方案A(MVP) | 方案B(推荐) | 方案C(旗舰) |
   |------|-----------|------------|------------|
   | 开发时间 | 1-2小时 | 4-8小时 | 2-5天 |
   | 功能完整度 | 60% | 85% | 100% |
   | 可扩展性 | 低 | 中 | 高 |
   | 维护成本 | 低 | 中 | 高 |
   | 风险 | 低 | 低 | 中 |
   ```

**Claude操作**:
- 基于探索结果生成多个方案
- 使用 `AskUserQuestion` 让用户选择方案
- 记录最终选择理由

### Phase 4: 快速原型 (Prototyping) ⚡

**目标**: 实现最小可行版本并演示

**执行步骤**:

1. **实现MVP**:
   ```python
   # Claude自动执行:
   1. 创建核心文件
      - Write tool
   2. 实现核心功能
      - 遵循CLAUDE.md规范
   3. 准备测试数据
      - 创建样例数据
   4. 编写测试脚本
      - 验证核心流程
   ```

2. **输出可运行代码**:
   ```markdown
   ## ⚡ MVP已完成

   ### 实现内容
   ✅ [功能1]
   ✅ [功能2]
   ✅ [功能3]

   ### 文件清单
   - [文件1]: [说明]
   - [文件2]: [说明]

   ### 如何测试
   ```bash
   # 测试命令
   python [测试脚本]
   ```

   ### 演示数据
   - [输入样例]
   - [输出样例]
   ```

3. **快速迭代循环**:
   ```
   用户测试 → 发现问题 → Claude立即修复 → 再次测试
   ```

**Claude操作**:
- 使用 `Write` 创建新文件
- 使用 `Edit` 修改现有文件
- 使用 `Bash` 运行测试
- 实时响应用户反馈

---

### Phase 5: 验证与迭代 (Validation) ✅

**目标**: 收集反馈并持续优化

**执行步骤**:

1. **收集反馈**:
   ```markdown
   ## 🔄 验证清单

   请您测试MVP并反馈:

   ✅ 功能是否符合预期?
   ✅ 是否满足核心需求?
   ✅ 还有什么需要调整的?
   ✅ 继续完善还是换个方向?

   您的反馈:
   [用户输入]
   ```

2. **决策分支**:
   ```
   如果反馈正面:
   → 继续完善功能
   → 集成到主系统
   → 更新文档

   如果反馈负面:
   → 分析问题原因
   → 调整方案
   → 重新验证

   如果需要转向:
   → 记录学习
   → 生成新方案
   → 重新开始Phase 3
   ```

3. **版本管理**:
   ```python
   # Claude自动执行:
   - 更新 version_registry.json
   - 记录到 evolution_log.json
   - 归档旧代码(如有)
   - 更新CLAUDE.md
   ```

**Claude操作**:
- 使用 `AskUserQuestion` 收集反馈
- 根据反馈调整代码
- 调用 `version_manager.py` 记录版本

## 最佳实践

1. **保持开放对话**
   - 不要急于求成

   - 理解真实需求

2. **快速试错**
   - 优先实现MVP

   - 迭代胜过完美

3. **记录一切**
   - 每个想法都值得记录

   - 形成组织知识

4. **用户驱动**
   - 用户决定方向

   - 快速响应反馈
